# Aizorix Services — Layout & Conventions

Every backend bounded context is an independent **Go module** under `services/<svc>/`,
wired together for local development by the workspace file at the repo root
([`../go.work`](../go.work)). Each service owns its own tables (DDL in
[`../db/migrations/`](../db/migrations)), exposes a REST surface via
[`go-chi/chi/v5`](https://github.com/go-chi/chi), talks to PostgreSQL via
[`jackc/pgx/v5`](https://github.com/jackc/pgx), and emits domain events through the
**transactional outbox** so a state change and the event it produces commit atomically.

The reference implementation is [`auth`](./auth); the shared "paved road" libraries live in
[`pkg`](./pkg). New services should mirror auth's shape exactly.

## Standard module layout

```
services/<svc>/
├── go.mod                       # module github.com/aizorix/platform/<svc>  (replace pkg => ../pkg)
├── go.sum                       # generated by `go mod download` / build
├── Dockerfile                   # multi-stage, distroless/static, nonroot
├── cmd/server/main.go           # boot: config → logger → pg pool → http server (graceful shutdown)
└── internal/
    ├── store/store.go           # pgx data access for THIS service's tables
    ├── service/service.go       # business logic + outbox event emission (tx-per-operation)
    └── httpapi/handlers.go      # chi routes + handlers + /healthz + /metrics
```

`internal/` keeps the store/service/httpapi packages unimportable by other modules — the
only supported cross-service contracts are HTTP/gRPC and Kafka events, never Go imports.

## The shared platform module (`pkg`)

`github.com/aizorix/platform/pkg` is the single dependency every service shares. Resolved
locally via `replace github.com/aizorix/platform/pkg => ../pkg` + the workspace.

| Package | What you use |
|---------|--------------|
| `config` | `LoadBase() Base`, `Get`, `GetInt`, `GetDuration` — env-driven config with defaults |
| `log`    | `New(level, service, env) *slog.Logger` — structured JSON logs with trace correlation |
| `pg`     | `Connect(ctx, url) (*pgxpool.Pool, error)` — pooled pgx with platform defaults |
| `outbox` | `Enqueue(ctx, tx, Event)` inside your tx; a separate `Relay` deployment publishes to Kafka |
| `rbac`   | `Principal{UserID, Roles, Permissions, AccountType}` with `Can` / `Require` / `HasRole`; `RequireOneOf` for ABAC ownership guards |
| `token`  | `Issuer` / `Verifier` for ES256 JWTs (auth issues; every service can verify locally) |
| `crypto` | Argon2 password hashing + envelope encryption for PII at rest |

## Conventions

- **Boot sequence** (`cmd/server/main.go`): `config.LoadBase()` → `log.New()` →
  `pg.Connect()` → construct `service.New(store.New(pool), …)` → start an `http.Server`
  with read/write/idle timeouts → block on `signal.NotifyContext(SIGINT, SIGTERM)` →
  `srv.Shutdown` with a 10s deadline.
- **Identity** is injected by the API gateway as request headers after it verifies the JWT:
  `X-User-Id`, `X-User-Roles` (comma-separated), `X-Account-Type`. Handlers build an
  `rbac.Principal` from these; they do not re-verify tokens on the hot path.
- **`/healthz`** returns `200` for liveness/readiness probes. **`/metrics`** exposes a
  Prometheus text exposition (`<svc>_up 1`) — swap in the prometheus client when wiring
  real metrics; the endpoint contract stays.
- **Data access** (`store`): `type Store struct{ pool *pgxpool.Pool }`, a `New(pool)`
  constructor, a `Pool()` accessor for transactions spanning store + outbox, a package
  `ErrNotFound`, and `pgx.ErrNoRows` mapped to it. Money is **`BIGINT` minor units (cents)**
  plus a currency code — never floats. Nullable columns scan into pointer types
  (`*int64`, `*string`, `*time.Time`); `text[]`/`uuid[]` map to Go `[]string`; `jsonb`
  is marshaled via `encoding/json`.
- **Business logic** (`service`): one operation = one transaction.
  `tx := pool.Begin(ctx)` → `defer tx.Rollback(ctx)` → store writes → `outbox.Enqueue(ctx,
  tx, Event{…})` → `tx.Commit(ctx)`. Sentinel errors (`ErrInvalidState`, `ErrForbidden`,
  …) are defined here and mapped to HTTP status codes in the transport layer.
- **Events**: name them `<aggregate>.<pastTense>` (e.g. `project.published`,
  `milestone.approved`), publish to a per-domain topic (`<aggregate>.events`), and set
  `PartitionKey` to the aggregate id so a single aggregate's events stay ordered (this requires a
  **single relay replica per source DB** — `SKIP LOCKED` across replicas does not preserve
  per-aggregate order; shard by `hash(partition_key)` to scale out).
- **Outbox relay** runs as a **separate deployment** against each service DB; the API pod
  does not run it (keeps responsibilities — and failure domains — clean).

| Service | Owns (core tables) | Emits |
|---------|--------------------|-------|
| `user` | freelancer/client profiles, skills, portfolio, KYC | `profile.updated`, `user.kyc_updated` |
| `project` | projects, categories, project skills/attachments | `project.published`, `project.closed` |
| `proposal` | proposals, milestones, answers | `proposal.submitted`, `proposal.withdrawn`, `proposal.shortlisted` |

| `review` | reviews (double-blind), reputation scores | `review.published` |
| `messaging` | conversations, messages, read receipts | `message.sent` |
| `notification` | notifications, channel prefs, delivery | — (consumer) |
| `search` | OpenSearch query API + indexers | — (consumer) |

## Adding a new service

1. **Schema first.** Add the service's tables (plus its own `outbox` table) in a new
   `db/migrations/0000NN_<svc>.up.sql` / `.down.sql`. The service owns those tables; it
   never writes another service's tables in production (cross-context calls go via API/event).
2. **Scaffold the module:**
   ```
   services/<svc>/
     go.mod   cmd/server/main.go   internal/store/store.go
     internal/service/service.go   internal/httpapi/handlers.go   Dockerfile
   ```
   Copy `go.mod` and `Dockerfile` from any existing service and replace the service name.
   The `go.mod` module path is `github.com/aizorix/platform/<svc>`, `go 1.22`, and requires
   `pkg` + `chi` + `pgx` with `replace github.com/aizorix/platform/pkg => ../pkg`.
3. **Register it in the workspace.** Add `./services/<svc>` to the `use (...)` block in
   [`../go.work`](../go.work).
4. **Implement** store → service → httpapi following the conventions above, then:
   ```bash
   cd services/<svc> && GOWORK=off go build ./...   # isolated module compiles
   go build ./...                                    # workspace-mode (cross-module) compiles
   ```
5. **Build the image** from the repo root (the build context is the repo root so the shared
   `pkg` module is in scope):
   ```bash
   docker build -f services/<svc>/Dockerfile -t aizorix/<svc>:dev .
   ```
